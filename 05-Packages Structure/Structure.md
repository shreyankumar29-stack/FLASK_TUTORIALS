# Flask Package Structure — `flaskblog` Project

## Overview

This document explains the folder structure of the **flaskblog** project — a Flask application built using the **application factory pattern**, where the app is organized as a Python package (`flaskblog/`) instead of a single script.

```
flaskblog/
├── __pycache__/
│   ├── __init__.cpython-314.pyc
│   ├── forms.cpython-314.pyc
│   ├── models.cpython-314.pyc
│   └── routes.cpython-314.pyc
├── instance/
│   └── site.db
├── static/
│   └── main.css
├── templates/
│   ├── about.html
│   ├── home.html
│   ├── layout.html
│   ├── login.html
│   └── register.html
├── __init__.py
├── forms.py
├── models.py
├── routes.py
├── run.py
├── Structure.md
venv/
├── Include/
├── Lib/
├── Scripts/
├── .gitignore
├── pyvenv.cfg
Readme.md
```

---
## Why Structure Matters

A small Flask app can live in a single app.py file, but as a project grows — routes, database models, forms, templates, static files — cramming everything into one file becomes unmanageable. Organizing your project into a proper package structure makes it:


Easier to navigate and maintain
Easier to test individual pieces
Easier to scale (add new features without touching unrelated code)
The standard expected by most real-world Flask projects and tutorials

## Folder-by-Folder / File-by-File Explanation

### `flaskblog/` (the main package)

This is the Python package containing your actual application code. Because it has an `__init__.py`, Python treats it as an importable package (`import flaskblog`).

#### `__pycache__/`
Auto-generated by Python whenever `.py` files are imported. Contains compiled bytecode (`.pyc` files) to speed up future imports. **This folder should never be committed to GitHub** — it's machine-generated and version-specific (notice `cpython-314`, tied to your Python version).

#### `instance/`
Flask's special folder for files that are **instance-specific** and shouldn't be in version control — like local config or a local database.
- `site.db` — your SQLite database file. Since it's inside `instance/`, Flask can access it without you specifying an absolute path, using `app.instance_path`.

#### `static/`
Holds static assets served directly to the browser — CSS, JavaScript, images.
- `main.css` — your stylesheet, linked in templates via:
  ```html
  <link rel="stylesheet" href="{{ url_for('static', filename='main.css') }}">
  ```

#### `templates/`
Jinja2 HTML templates rendered by your routes.
- `layout.html` — the **base template** (navbar, footer, common structure) that other pages extend using `{% extends "layout.html" %}`.
- `home.html` — the blog's home/feed page.
- `about.html` — a static about page.
- `login.html` — login form page.
- `register.html` — user registration form page.

#### `__init__.py`
The **application factory** — this is where the Flask app is created and configured. Typically looks like:
```python
from flask import Flask

app = Flask(__name__)
app.config['SECRET_KEY'] = 'your-secret-key'
app.config['SQLALCHEMY_DATABASE_URI'] = 'sqlite:///site.db'

from flaskblog import routes
```
Because this file exists, `flaskblog` is treated as a package, and `from flaskblog import app` works from `run.py`.

#### `forms.py`
Contains **Flask-WTF form classes** — e.g., `RegistrationForm`, `LoginForm` — used to render and validate the login/register templates.
```python
from flask_wtf import FlaskForm
from wtforms import StringField, PasswordField, SubmitField

class LoginForm(FlaskForm):
    email = StringField('Email')
    password = PasswordField('Password')
    submit = SubmitField('Login')
```

#### `models.py`
Defines your **database models** using SQLAlchemy — the tables that get stored in `instance/site.db`.
```python
from flaskblog import db

class User(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    username = db.Column(db.String(20), unique=True, nullable=False)
    email = db.Column(db.String(120), unique=True, nullable=False)
    password = db.Column(db.String(60), nullable=False)
```

#### `routes.py`
Contains all your **view functions / route handlers** — the logic connecting URLs to templates.
```python
from flask import render_template
from flaskblog import app

@app.route('/')
@app.route('/home')
def home():
    return render_template('home.html')

@app.route('/about')
def about():
    return render_template('about.html')
```

#### `run.py`
The **entry point** of the whole project — the file you actually execute to start the server. Kept outside the package so it just imports and runs the app.
```python
from flaskblog import app

if __name__ == '__main__':
    app.run(debug=True)
```

#### `Structure.md`
A markdown file (like this one) documenting the project's folder structure for anyone browsing the repo.

---

### `venv/` (virtual environment — outside the package)

This is your Python **virtual environment**, isolating this project's dependencies from your global Python installation.
- `Include/`, `Lib/`, `Scripts/` — internal virtual environment files (installed packages, activation scripts, Python binaries).
- **This entire folder should be excluded from GitHub** via `.gitignore` — it's large, machine-specific, and easily recreated with `pip install -r requirements.txt`.

### `.gitignore`
Tells Git which files/folders to ignore when committing. For this project, it should include at minimum:
```
venv/
__pycache__/
*.pyc
instance/
.env
```

### `pyvenv.cfg`
Auto-generated config file inside the virtual environment recording which Python version created it. You never edit this manually.

### `Readme.md`
The top-level repo documentation — describes what the project is, how to install/run it, and any other details for visitors to your GitHub repo.

## Why This Structure Works

| Principle | How this structure achieves it |
|---|---|
| **Separation of concerns** | Models, forms, and routes are in separate files instead of one giant script |
| **Package-based app** | `flaskblog/__init__.py` makes the app importable and testable |
| **Clean entry point** | `run.py` sits outside the package, just starting the server |
| **Local-only files excluded** | `instance/`, `venv/`, `__pycache__/` are correctly separated from source code |
| **Static/template separation** | Follows Flask's default convention so no extra config is needed |

```

flaskblog/__pycache__/
flaskblog/instance/
*.pyc
.env
```

To generate `requirements.txt` before pushing:
```powershell
pip freeze > requirements.txt
```

## Summary

The `flaskblog` project follows Flask's standard **package-based structure**: application code lives inside the `flaskblog/` package (with `__init__.py`, `models.py`, `forms.py`, `routes.py`), templates and static files sit alongside it, and `run.py` acts as the external entry point. The `venv/` folder and `instance/` database are correctly kept separate from version-controlled code — exactly what should be excluded from GitHub via `.gitignore`.
